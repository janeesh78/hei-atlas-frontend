# Hei Atlas — Engineering Overview

A voice-first ambient documentation and clinical decision support web app for
oncology clinics. The physician records the patient encounter through the
**ATLAS** ambient-listening control, the audio is transcribed, a structured
oncology note is generated by an LLM, and downstream analyzers surface
toxicity grading, billing/coding intelligence, guideline citations, and a
one-click clinical-trial matcher.

This document is intended as a high-level technical orientation. Internal
heuristics, prompt engineering, regex/vocabulary catalogs, model selection
logic, and other proprietary implementation details are not described here.

## Pipeline (single encounter, end-to-end)

```
1.  ATLAS recording in the browser (MediaRecorder)
       │
       ▼
2.  POST /transcription/    ← multipart audio
       │   primary engine + a secondary engine fallback
       │   audio decoding bundled with the backend (no system dependency)
       │   output post-processed for quality before being returned
       ▼
3.  Intent classification (frontend) routes the payload to one of:
       │     • note generation pipeline (default for dictated encounters)
       │     • clinical-trial search (for short typed queries)
       │     • guideline / general query
       ▼
4.  POST /notes/generate    ← { transcript, output_format, previous_note? }
       │   LLM produces a structured OncologyNote
       │
       ├─→ Toxicity analyzer (CTCAE v5)        — structured AE grading
       ├─→ Coding intelligence (CMS 2021/2023) — E/M, ICD-10, CPT, gaps
       └─→ Guideline citation matcher          — context-aware references
       │
       ▼
5.  On-demand: GET /cds/trials   ← ClinicalTrials.gov v2 (sanitized query)
```

Every frontend pipeline step posts a small telemetry marker to a backend
debug endpoint so silent failures can be reconstructed from the server log
without requiring access to the user's browser console.

server I/O; each panel is pure-presentation and receives props.

## Cross-cutting concerns

| Concern | Approach |
|---|---|
| **PHI on the wire** | Every outbound third-party request is filtered through a sanitization layer in both the frontend client and backend services. Free-text input is reduced to safe, bounded clinical vocabulary before crossing the wire. |
| **Telemetry** | Each pipeline step on the frontend posts a small marker to a backend debug endpoint. Used to reconstruct failures without browser-side access. |
| **Auditability** | Request middleware writes a structured audit log per request. Coding output, citations, and toxicity findings are tagged with a versioned engine identifier. |
| **Versioning** | The coding engine, CTCAE schema, and frontend pipeline each carry an explicit version constant. Future revisions are additive. |
| **Failure isolation** | Third-party calls (transcription fallback, literature search, trial search) are wrapped so that one failing service degrades gracefully without aborting the whole encounter. |
| **Clipboard hygiene** | All copy / download / print output passes through a sanitizer to ensure pasted clinical text remains plain ASCII across destination editors (Epic, Word, etc.). |

## Design system (Tailwind tokens)

| Token | Purpose |
|---|---|
| `canvas` / `surface` / `ink` / `muted` / `rule` | Background, card, primary text, secondary text, dividers |
| `accent` / `accent-subtle` / `accent-hover` | Brand color (active states, focus rings, primary actions) |
| `card` / `button` / `pill` radii | Consistent corner radii across surfaces |
| Inter | Primary font, with system fallback for first-paint |

Three-column layout collapses progressively: left sidebar hides below the
large breakpoint, right panel hides below the extra-large breakpoint.

## Local development

```bash
scripts/start-local.sh           # bring up Docker, DB, Redis, backend, frontend
scripts/start-local.sh status    # health check
scripts/start-local.sh stop      # tear down
```

The backend reads an `.env` file for third-party API keys; the start script
loads it automatically.
